# T2.1 — Keystone History Slot

## Problem

The conversation history slot uses pure recency ordering: the most recent N messages fill the budget. This means high-signal older messages (architecture decisions, deployment milestones, incident resolutions) age out when the conversation gets long. The agent loses institutional memory within a single session.

## Approach

Split the history budget into two sub-allocations:

1. **Recent history** (70-80%): most recent messages, as today
2. **Keystone history** (20-30%): high-signal older messages selected by episode significance + FTS relevance to the current query

The keystone selection pipeline:

```
  All history messages (Redis)
         │
    ┌────┴────────────────┐
    │  Recent (80%)       │  ← newest messages up to budget
    │  unchanged logic    │
    └─────────────────────┘
    ┌────┴────────────────┐
    │  Keystone (20%)     │  ← selected from OUTSIDE the recent window
    │  scored by:         │
    │   - episode sig     │  ← from episodes table, joined on msg ref
    │   - FTS relevance   │  ← current-turn query matched against older msgs
    │   - recency boost   │  ← gentle penalty for very old messages
    └─────────────────────┘
```

## Keystone Scoring Formula

```
keystone_score = (episode_significance * 0.5)
              + (fts_relevance * 0.3)
              + (recency_factor * 0.2)

where:
  episode_significance = significance field from episodes table (0.0 - 1.0)
                         or 0.3 default if message not linked to any episode
  fts_relevance       = normalized BM25 rank from FTS5 match against current query
                         0.0 if no match
  recency_factor      = 1.0 - (message_age_hours / max_history_age_hours)
                         clamped to [0.0, 1.0]
```

## Data Sources

### Episode linkage
Episodes already store `session_key` and we have `source_ref: 'msg:{id}'` on facts. For episodes, we currently store a `summary` which is the first 200 chars of the message content.

**Gap:** Episodes don't currently store `source_message_id`. We need to add this. 

Schema migration:
```sql
ALTER TABLE episodes ADD COLUMN source_message_id INTEGER;
CREATE INDEX idx_episodes_source_msg ON episodes(agent_id, source_message_id);
```

The indexer's `classifyEpisode` call needs to pass `msg.id` through so `record()` can store it.

### FTS relevance for history messages
History messages are in the per-agent messages.db, not in library.db. We have two options:

**Option A: FTS5 index on messages table** (preferred)
Add an FTS5 index to `messages.text_content` in messages.db. This is the most direct path — query old messages by relevance to the current prompt. Cost: FTS5 index on messages.db (already has 12k rows for forge alone). Runtime: one FTS5 query per compose, bounded by LIMIT.

**Option B: Use episode summaries as proxy**
The episodes table already has FTS5. Match current query against episode summaries, then retrieve the linked messages. Less precise but no new index needed.

Recommendation: **Option A** for precision, with Option B as a zero-new-infrastructure fallback.

## Compositor Integration

In `Compositor.compose()`, after the existing history assembly:

1. Calculate budget split: `keystoneBudget = Math.floor(historyTokens * 0.2)`
2. Reduce recent history budget: `recentBudget = historyTokens - keystoneBudget`
3. After assembling recent history, determine the oldest message ID in the recent window
4. Query keystone candidates from messages older than that ID:
   - Join with episodes (on source_message_id) for significance
   - FTS5 match against current prompt for relevance
   - Score and rank
5. Select top-K keystones that fit within keystoneBudget
6. Insert keystones chronologically into the message array, marked with a `[keystone]` metadata tag
7. Add a separator comment so the model knows these are out-of-sequence recalled messages

### Message format in context

```
## Recalled Context (high-signal older messages)

[current messages in recency order]
```

## Budget Impact

With the TUNE-010 90k budget and ~5k system overhead:
- History gets ~45k tokens (50% of budget via historyDepth)
- Keystone slice: ~9k tokens (20% of 45k)
- Recent history: ~36k tokens
- At ~500 tokens/message avg: ~18 keystone messages possible, ~72 recent messages

In practice, keystones are shorter (summaries) so we'll get 10-15 high-signal recalled messages plus the full recent conversation.


## Implementation Steps

1. **Schema migration**: Add `source_message_id` to episodes table
2. **Indexer update**: Pass message ID through episode classification and recording
3. **Messages FTS5 index**: Add FTS5 virtual table on messages.text_content (Option A)
4. **Keystone scorer**: New function `scoreKeystoneCandidate(message, episode, ftsRank, maxAge) → number`
5. **Compositor integration**: Split history budget, query keystones, inject into message array
6. **Config**: Add keystone config fields with defaults
7. **Tests**: Compositor test that verifies keystone messages appear in output for long conversations

## Dependencies

- TUNE-011 (indexer quality) should land first — keystone selection depends on episode quality
- Vector store wiring (separate track) would improve keystone selection via semantic matching, but FTS5 is sufficient for v1

## Risk

- **Over-injection**: Too many keystones might confuse the model with context-switching. Cap at 15 messages and monitor.
- **FTS noise**: Some old messages might match current terms but be irrelevant. Episode significance weighting helps — pure FTS matches without episode linkage score lower.
- **Budget pressure**: 20% of history is meaningful. If conversations are short (<20 messages), keystones add nothing. Gate: only activate keystone retrieval when recent history exceeds 30 messages.
