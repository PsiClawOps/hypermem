# T2.2 — Vector Store Wiring

## Problem

The VectorStore code is fully implemented and tested. sqlite-vec is installed and works.
Ollama + nomic-embed-text is running locally. But HyperMem.create() never initializes a
VectorStore, so the compositor runs in FTS5-only mode with no semantic similarity.

## Current State

- `VectorStore` class: ✅ complete, tested in test/vector-search.mjs
- `sqlite-vec` npm package: ✅ installed in hypermem/node_modules
- `sqlite-vec` extension: ✅ loads and works (confirmed with manual test)
- `nomic-embed-text` on Ollama: ✅ running on localhost:11434
- `hybridSearch()`: ✅ handles vectorStore=null gracefully (FTS5-only fallback)
- `Compositor`: ✅ accepts vectorStore in constructor deps
- `HyperMem.create()`: ❌ sets vectorStore=null and never updates it
- Plugin init: ❌ doesn't create or wire a VectorStore

## What Needs to Change

### 1. HyperMem.create() — Wire vector store

After creating the instance, initialize VectorStore:

```typescript
// In HyperMem.create(), after Redis connect:

// Initialize vector store with sqlite-vec
try {
  const vectorDb = hm.dbManager.getOrCreateVectorDb();  // New method
  const vs = new VectorStore(vectorDb, merged.embedding, hm.dbManager.getLibraryDb());
  vs.ensureTables();
  hm.compositor.setVectorStore(vs);  // New setter
  console.log('[hypermem] Vector store initialized');
} catch (err) {
  console.warn('[hypermem] Vector store unavailable (sqlite-vec not loaded?):', (err as Error).message);
}
```

### 2. DatabaseManager — Vector DB management

Add `getOrCreateVectorDb()` method that:
1. Opens/creates `{dataDir}/vectors.db`
2. Loads sqlite-vec extension
3. Returns the DatabaseSync instance

```typescript
getOrCreateVectorDb(): DatabaseSync {
  if (this.vectorDb) return this.vectorDb;
  const dbPath = path.join(this.dataDir, 'vectors.db');
  const db = new DatabaseSync(dbPath, { allowExtension: true });
  db.enableLoadExtension(true);
  
  // Load sqlite-vec
  const sqliteVec = require('sqlite-vec');
  sqliteVec.load(db);
  
  this.vectorDb = db;
  return db;
}
```

### 3. Compositor — VectorStore setter

Add a public setter since vectorStore is set after construction:

```typescript
setVectorStore(vs: VectorStore): void {
  (this as any).vectorStore = vs;
}
```

### 4. Background Indexer — Embedding pipeline

When processing messages, if a VectorStore is available, embed new facts and episodes:

```typescript
// After storing a fact:
if (vectorStore) {
  await vectorStore.indexItem('facts', factId, factContent, domain);
}
```

This can run asynchronously — embedding is best-effort. Failures shouldn't block fact extraction.

### 5. Initial bulk embedding

One-time migration script to embed all existing facts/episodes:

```typescript
// scripts/embed-existing.mjs
// Read all un-embedded facts from library.db
// Batch embed via Ollama (32 at a time)
// Store in vectors.db via VectorStore.indexItem()
```

With 2,527 facts and batch size 32, this is ~79 Ollama calls. At ~100ms each, ~8 seconds.

## Plugin Integration

The plugin needs to:
1. Pass embedding config through to HyperMem.create()
2. Ensure sqlite-vec is importable from plugin context

Since sqlite-vec is already a dependency of hypermem (not the plugin), the dynamic import
path should work if the plugin resolves to the hypermem dist.

## Risk

- **sqlite-vec loading**: Node.js SQLite requires `allowExtension: true` on database creation.
  Need to ensure DatabaseManager creates vector DB with this flag.
- **Ollama availability**: Embedding calls fail if Ollama is down. VectorStore handles this
  gracefully — FTS5 continues working. But we should log, not crash.
- **Memory**: 768-dim float32 vectors × 2,527 facts = ~7.5MB. Negligible.
- **Latency**: Embedding one fact (~100ms) during indexer tick is fine. Embedding at compose
  time (for semantic recall query) adds ~100ms to context assembly. Acceptable.
